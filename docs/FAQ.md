# FAQ

Questions that came up while building and reasoning about this framework, kept here because the answers involve context that doesn't fit naturally into `README.md` or the other reference docs. For the framework's actual hard rules (what it will never do, and why) rather than the reasoning behind them, see `docs/CONSTRAINTS.md`.

## Do I have to use Claude specifically?

No. The folder + `CONTEXT.md` mechanism is agent-agnostic — any LLM agent that respects a stage's declared input scope can run it. `docs/CLAUDE_WORKFLOW_PLAYBOOK.md` is a reference workflow for teams using Claude (which Chat/Cowork/Code CLI surface fits which stage), not a requirement. Swap in a different agent and the rest of the framework works unchanged.

## Can I use this framework to develop the framework itself?

Yes, with one boundary: this repo is meant to stay a clean, product-agnostic template (`features/` and `sprints/` are gitignored so no live workspace data accumulates here). So "self-hosting" happens one of two ways:

- **Most changes** (a `CONTEXT.md` tweak, a script fix, a doc update): use `docs/evolution/` directly, in this repo. Write an entry, edit the actual file in the same commit, mark it `adopted`.
- **Bigger, genuinely uncertain structural changes** (e.g. testing a new scoring model): pilot it in a separate sandbox workspace using `scaffold.sh --feature` against a copy of the templates, run it through a real Build-Measure-Learn cycle, then fold only the validated result back into this repo as an evolution entry. Never commit the scaffolded feature folder itself here.

For the mechanics of actually making a change — script conventions, which doc file something belongs in, the checklist of what to update on every adoption, and which backlog entries currently touch the same files — see `docs/DEVELOPMENT.md`. This FAQ answer covers *whether/when* to self-host; that file covers *how*.

## Is developing the framework itself a PDLC process?

Yes — concretely, not just by analogy. The session that produced v1 of this framework maps onto all 6 stages: the original brainstorm and grounding it against reality (01), the design decisions that got locked in (02), the stage contracts and templates (03), the folder/script architecture (04), writing and actually testing the scripts end-to-end (05), and publishing to GitHub (06, in progress — the "go-to-market" half). The one thing not yet done is the "validation" half of stage 06: running a real product feature through the framework to test whether folder isolation and `CONTEXT.md` contracts are actually sufficient without a heavier orchestration layer. That's the framework's own riskiest assumption, and it's still open — see `docs/evolution/0001-first-principles-analysis.md`'s trigger conditions.

## Is `docs/evolution/` part of the framework, or something separate?

Both — it's a shipped, committed part of the repo (listed in `README.md`'s file map), but it's a parallel track to the 6-stage product pipeline, not a stage inside it. The 6 numbered stages are how a *product built with this framework* moves from assumption to validated ship. `docs/evolution/` is how the *framework itself* moves from assumption to validated change — deliberately lighter-weight, since framework tweaks are usually lower-stakes than shipping a product feature. Entries there (like 0002–0006) are shaped like Lean discovery candidates — a stated problem, a proposed fix, a stepwise plan — but they aren't literal `01_discovery_ideation` stage outputs from a scaffolded feature folder, because nobody ran `scaffold.sh --feature` for them.

## Why isn't the C-V-R prioritization formula used for framework-evolution entries?

C-V-R scores product features against business/engineering risk. Evolution entries are scored informally by priority rank (see each entry's "Priority: N of 5") based on how much real friction they remove versus how speculative they are — closer to expected-value-of-information than to C-V-R's specific formula. If this framework migrates toward the Decision Node model described in entry 0001, both product features and framework-evolution entries would likely score the same way.

## Does the framework have any stated rule against bloat or vendor lock-in, or is that just an informal habit?

It's now a stated rule, as of entry `0031`. It was always followed in practice — every "no auto-install," "no configurable logging," "no alerting" call in `docs/CONSTRAINTS.md` is really the same underlying judgment (don't add machinery without a concrete need), and the framework has stayed agent-agnostic and format-agnostic throughout (plain markdown, any LLM agent per the folder/`CONTEXT.md` contract) — but none of that was written down as a checkable principle until it got applied to something new (see the next entry) and the reasoning had to be reconstructed from scratch in the moment. `docs/CONSTRAINTS.md`'s new "Extensibility & composability" section now states it directly: prefer a separate tool/plugin over folding a capability into the framework's core when it would only help a subset of products, and no proposal may hard-couple the framework to one vendor, model, or proprietary format. Both restate truths entry `0001` already identified — this just makes them reviewable against a written rule instead of pattern-matched from precedent. See `docs/evolution/0031-composability-anti-lock-in-principle.md`.

## Does the framework support OKF (Open Knowledge Format), and should it?

No, and not planned for now. OKF is Google's spec for packaging RAG knowledge bases (documents, chunks, embeddings, a manifest) — published 2026-06-12, under a month old when this was surveyed. This framework's docs are plain markdown with none of that structure. The bigger reason to hold off isn't the format's age, though — OKF solves semantic search over large document corpora, and this framework's `CONTEXT.md`-declared-scope model doesn't have that problem (an agent is told exactly which files to read; there's no retrieval step to standardize). Adopting OKF's chunking/embedding machinery now would be new coupling to an unproven, single-vendor format, which is exactly what the previous FAQ entry's new rule exists to catch. Revisit if OKF gains real multi-vendor adoption, or a product built from this framework has a concrete need to export its docs into an external tool that requires it specifically. See `docs/evolution/0032-deferred-survey-okf-performance-pilot.md`.

## Is framework performance tracked, or is an end-to-end pilot test coming?

Neither yet, and both are deliberately deferred rather than forgotten. Performance tracking: the framework has no runtime to benchmark in the usual sense (no server, nothing to measure latency/throughput on) — the one thing that would actually be meaningful, lead time from `scaffold.sh` to stage-06 completion, is already computable from `SYNC_LOG.md` timestamps without new instrumentation, but there's no real data to trend yet since no pilot has completed a cycle. Building that before there's a single data point would be measuring nothing. End-to-end pilot: this is the framework's own riskiest assumption (see entry `0001`'s trigger conditions and README's "Status" section) and it genuinely can't be resolved by another evolution entry — `CLAUDE.md` and `README.md` both forbid building product data inside this repo, so there's no scaffolding change that substitutes for a real product actually running through all 6 stages somewhere else. Both stay open until that happens. See `docs/evolution/0032-deferred-survey-okf-performance-pilot.md`.

## Can I just build my product directly inside this framework repo instead of copying it elsewhere?

Don't. Nothing technically stops you, but `features/` and `sprints/` are gitignored specifically so this repo stays reusable across every future product. Build directly here and you lose the ability to cleanly reuse the framework next time — you'd have to manually pick the generic parts back out of a repo full of one product's history. Always copy `.mwp-templates/`, `scripts/`, and `CLAUDE.md` into a new, separate product repo, per the README's "Using this framework for a new product" steps.

## Should I let an AI agent (Claude Code, Cowork, etc.) run git commands directly against my working folder?

Be cautious about this, especially since Claude Code CLI is one of the three surfaces this framework's own playbook recommends for stages 05–06. While building this repo, running `git init`/`git commit` through a sandboxed AI environment against a folder synced from a real disk caused real corruption — `.git/config` came back zero-filled, and stale lock files (`index.lock`) were left behind, blocking all future git operations until manually cleared. The sandbox's file-sync mechanism didn't handle git's atomic rename/lock-file writes correctly, even though plain file reads/writes/deletes worked fine.

The safe pattern that worked: do git operations (`init`, `add`, `commit`, `push`) from your own terminal, on your own machine, using your own git installation. If an agent needs to prepare a repo on your behalf, it should build it in an isolated scratch location first and copy only the *finished* `.git` directory over as a whole-file copy — never run git commands incrementally against a live, synced working folder.

**So can Claude just add/commit/push for you directly, going forward?** Technically it could attempt to, but given the corruption incident above, the established pattern for this repo is: Claude edits files and hands you the exact `git add`/`commit`/`push` commands to run yourself, every time — never runs git against this live working folder on your behalf. This isn't a hard technical limitation, it's a deliberate policy adopted after a real incident, and it's held for every adoption in this evolution log so far.

## I just watched Claude edit a script, then test it in its sandbox and get stale/truncated results — what happened?

A related but distinct issue from the git corruption above, discovered while testing entry `0003`'s `scaffold.sh` change: after editing the file, copying it (`cp`) from the live mounted repo path into the sandbox served a version missing the last few lines, even though the actual file on disk was already correct and complete — confirmed by reading it back directly. Re-copying didn't fix it either; the mount stayed stale for the rest of that session. This is a read-side staleness issue (a plain file read returning old content), not the write-side corruption the git entry above describes, but it's the same underlying caution: don't fully trust the sandbox-to-Windows mount mid-session for anything you're about to rely on.

The workaround that worked: write the file's exact current contents directly into the sandbox's own scratch space (not copied through the live mount) and test from there. If a test result from Claude's sandbox looks wrong immediately after an edit — truncated output, a script behaving like an older version — this mount staleness is worth suspecting before assuming the code itself is broken. See `docs/evolution/0003-computed-priority-registry.md`'s Outcome section.

**Not just tied to recent edits, either.** While testing entry `0006` (`--sprint` mode), `registry.sh` — a file untouched that entire session — came back truncated via a fresh `cp` from the mount, right after two other, recently-edited scripts had already hit the same issue. So the staleness isn't reliably correlated with "this file was just edited"; any file read through the live mount mid-session is worth verifying (`wc -l` / `tail` against what a direct `Read` already showed) before trusting a test result built on it, not just ones edited a moment ago.

**Also seen as mid-string truncation, not just missing trailing lines — and the "just write it to scratch space" fix has a gap.** While doing final verification for entry `0036`, the same class of bug showed up differently: `bash -n` on a mount-copied `scaffold.sh` failed with `unexpected EOF while looking for matching "'`, and `wc -l` confirmed the file was cut off mid-string, 7 lines short — not truncated at a clean line boundary. This reproduced across repeated `cp` attempts and even a fresh whole-repo copy. The bigger catch: the workaround described above ("write to the sandbox's own scratch space") isn't fully safe either if that scratch space is itself a Windows-mounted folder — which the `outputs` working directory is, in this setup. The fix that actually held: rebuild the file content into a path with **zero mount involvement at all** (e.g. `/tmp/...` in the sandbox), sourcing the content from the `Read` tool's output rather than any `cp`/`cat` through the mount, then run tests from there. `Read` has been reliable throughout every incident of this kind so far; only mount-mediated copies and executions have not. See `docs/evolution/0036-consistent-help-flags.md`'s Outcome section.

## Why couldn't Claude test `scripts/preflight.ps1` itself, the way it tested the bash scripts?

Claude's sandbox is Linux, with no PowerShell installed at all (confirmed: `pwsh` isn't available) — unlike the bash scripts, where the sandbox is at least a partial stand-in even though real verification still happens on the actual machine per entry `0002`. For a `.ps1` file there's no fallback: it can only be written, never run, inside this environment. Any PowerShell-only tooling this framework adds (entry `0023` is the first) will always need its actual testing done on a real Windows machine, with Claude only able to review the script by reading it and reasoning about correctness, not executing it.

Verifying entry `0023` on the real machine also surfaced a second limit worth knowing: `C:\Windows\System32\bash.exe` is a Windows-shipped WSL launcher stub that exists unconditionally on modern Windows, so "bash is completely absent from PATH" isn't really a simulatable state — trimming `PATH` down to just `System32` still finds it. The workaround was testing the same `Get-Command`/if-else/`exit 1` logic against a command guaranteed not to exist, rather than trying to hide bash itself. Worth remembering for testing any future PowerShell detection script: the "not found" branch may need a stand-in target, not a literal repro. See `docs/evolution/0023-powershell-bash-preflight-check.md`'s Outcome section.

## My first `git push` to a freshly created GitHub repo was rejected as "non-fast-forward" — why?

GitHub can add an initial commit (e.g. a license file) even when you think every "initialize this repository" checkbox was left unchecked — this happened during this framework's own setup. Check what's actually on the remote before assuming a real conflict:

```
git fetch origin
git show origin/main --stat
```

If it's just a placeholder file with nothing you need, it's safe to overwrite:

```
git push --force-with-lease --set-upstream origin main
```

`--force-with-lease` is safer than plain `--force` — it double-checks the remote hasn't changed unexpectedly since your last fetch before overwriting anything.

## `git status` shows dozens of files as modified right after a clean pull or checkout on Windows, even though I haven't touched them — why?

This happened in this repo's own history: after landing entries `0038`–`0041`, `git status` listed roughly 60 unrelated, untouched files (`CLAUDE.md`, `LICENSE`, most of `docs/evolution/`, several `scripts/lib/*.sh` and `tests/test_*.sh` files) as modified, and `git pull --rebase` refused to proceed because of them. The first two things that looked like the obvious cause weren't it: `git config core.autocrlf false` changed nothing, and `git restore .` changed nothing either — the exact same file list came back immediately after.

The actual cause only showed up by reading real diff content instead of just `--stat` counts (`git diff -- CLAUDE.md | head`): every flagged file's diff was purely `old mode 100755` / `new mode 100644` — a Unix executable-bit mismatch, not a content or line-ending change at all. This repo has a number of files committed with the Unix executable bit set; Git for Windows / MSYS2 doesn't reliably reproduce that bit on checkout, so `git status` compares the working tree's mode against the committed mode and reports every one of them as "modified" even though the bytes are identical. The CRLF warnings Git prints alongside (`warning: LF will be replaced by CRLF...`) are unrelated boilerplate in this case — easy to misread as the cause, since they show up on the same files.

Fix: `git config core.fileMode false` (local to the repo, not global) — tells git to stop comparing/tracking the executable bit for this repo entirely. `git status` clears immediately, with no `git restore` or re-clone needed. Worth knowing this is a distinct setting from `core.autocrlf`: autocrlf controls line-ending conversion, fileMode controls whether permission-bit changes count as a diff — fixing one doesn't touch the other, and it's easy to chase the wrong one first given how similar the symptoms look at a glance.

## Which docs travel with a new product built from this framework, and which stay behind?

**Travels:** `docs/FAQ.md`, `docs/CLAUDE_WORKFLOW_PLAYBOOK.md`, `docs/PRIORITIZATION_GUIDE.md`, `docs/TOOLING_MATRIX.md`, `docs/CONSTRAINTS.md` — these are reference material a product team actually needs day-to-day, regardless of which product they're building. The README's "Using this framework for a new product" copy commands include them.

**Stays behind:** `docs/evolution/` (this repo's own entries), `PROJECT_PLAN.md`, and `docs/DEVELOPMENT.md` — these are about *this framework template's own* design history and how to develop it (why it has 6 stages, why C-V-R has a bypass rule, why the scripts are shaped the way they are). None of that is about your product.

**Ships automatically, but starts empty:** your product gets its own `docs/evolution/EVOLUTION_LOG.md`, generated from `.mwp-templates/PRODUCT_EVOLUTION_LOG_TEMPLATE.md` as part of the same copy step — the append-only convention travels (see that file's "How to use this" section), the framework's own specific entries don't. This used to be a "start one yourself if you want it" suggestion; it's now handed to every new product with zero entries, ready to use from feature one. See `docs/evolution/0007-product-level-decision-log.md`.

## Can VS Code (or Claude Code CLI) automatically capture FAQ-worthy conversations into this file?

Yes, going forward — `CLAUDE.md`'s root instructions now tell Claude Code (CLI or the VS Code extension, both read the file the same way) to proactively append reusable Q&A to this file whenever a conversation in this repo resolves one, without waiting to be asked. This only fires while Claude Code is actually working inside this repo (or a product repo copied from it) — it doesn't reach into Claude Chat conversations or retroactively pull in past sessions. Anything from before this instruction existed, or from a different surface (Chat, a different project), still needs to be pasted in manually.

The instruction's wording was generalized after a gap was caught: it originally said "about how the framework works," which read fine here but could be misread narrowly once copied verbatim into a product repo — as "the ICM/MWP methodology" rather than "this product." It now says "about how this repo/system works — the framework's own mechanics if this is the framework template repo, or this product's architecture/behavior if this is a product workspace built from it," so genuinely reusable product-specific Q&A (e.g. "why does our auth flow work this way") gets captured too, not just framework-mechanics Q&A. ## How should I name and describe a repo built from this framework?

Lowercase, hyphenated (`your-product-name`, not `YourProductName`) — GitHub URLs are case-insensitive anyway, and it matches the near-universal convention plus any tooling that assumes lowercase paths. For the repo's top-line description, keep it agent-agnostic even if you use Claude day-to-day — the core folder/`CONTEXT.md` mechanism works with any LLM agent, and naming one product in the front-door pitch narrows your audience and can read like an implied endorsement you don't have. Save Claude-specific detail for `docs/CLAUDE_WORKFLOW_PLAYBOOK.md`, where it's accurately scoped to where it actually applies.

## When does the framework check whether a required tool (Fabric, Graphify, Repomix, DuckDB) is installed — and does it install missing ones automatically?

Two layers, once entry `0013` is implemented: an on-demand `doctor.sh` that scans the whole tool matrix at once (run manually, whenever), and a lazy per-stage check right before `CLAUDE.md`'s automation routing tries to invoke a tool — if it's missing, that check fails clearly via the existing escalation contract (`BLOCKED_REASON.md`) instead of a cryptic "command not found" mid-task. Nothing checks at `scaffold.sh` time, since a feature might not reach the stage that needs a given tool for a while.

No auto-install by default. Installing software is a bigger side effect than anything else this framework does — everything else is scoped to reading/writing markdown in a declared folder. A missing tool escalates with the exact install command; a human decides whether to run it. `doctor.sh --install-missing` exists as an explicit opt-in for anyone who wants the convenience, but it's never the default behavior. See `docs/evolution/0013-verify-tooling-matrix.md`.

## Does the framework support configurable logging (log levels, per-stage verbosity)?

No, deliberately. What exists is a single always-on operational log — one line per script invocation, no levels, no per-stage config — because a configurable logging subsystem is exactly the kind of machinery entry `0001`'s first-principles analysis already flagged as the "coordination requires software" dogma this framework rejects. It would also add more surface area that context-bundling tools (Repomix, Graphify) have to be told to ignore, and a shared config file editable by multiple features risks becoming the same class of coordination-cost problem entry `0003` is trying to fix for the priority registry. See `docs/evolution/0014-minimal-operational-log.md`.

## How do we come up with requirements for this framework, and is now a good time to do that deliberately?

Two different questions bundled together, worth separating.

*Where do requirements come from?* Looking back at all 30 entries logged so far, three channels have actually produced them: friction found while using the framework (entries `0025` and `0027` were both surfaced incidentally while testing something unrelated, not from deliberate requirements-gathering), first-principles critique (entry `0001`, deliberately stripping away assumptions to find what's load-bearing versus habit), and external research filtered for relevance (entries `0029`/`0030`). All three funnel into the same shape — a `docs/evolution/` entry's Problem/Proposed-change/Stepwise-plan is really stages 01/02/03 (discovery/definition/requirements) compressed into one file instead of three folders, the same self-hosting pattern described in "Can I use this framework to develop the framework itself?" above. Entry `0001` predicts this directly: it argues the real fundamental unit isn't a stage or a feature but a Decision Node (an assumption, the cost if wrong, the cheapest experiment to test it) — an evolution entry is basically a Decision Node made manual and lightweight, applied to the framework's own development instead of a product's.

*Is now a good time?* Check entry `0001`'s own stated trigger conditions for when this lightweight approach starts straining (the C-V-R bypass rule getting invoked often, a stage needing to reach further back than one step, two changes that should've been sequential colliding, fixed ceremony being skipped as disproportionate, needing per-feature Micro-vs-Agile choice across products). None have fired as of this entry — the "Known cross-entry collisions" list in `docs/DEVELOPMENT.md` is currently empty. So there's no evidence the current approach needs to become heavier. That said, a cleared backlog with nothing on fire is exactly the calm window where a deliberate discovery pass (rather than only reactive additions) tends to pay off — the same reasoning a product team would apply between sprints. The framework's own biggest open item either way is still what "Is developing the framework itself a PDLC process?" above flags: no real product has piloted it end-to-end yet, which is the actual riskiest assumption a deliberate requirements pass would need to weigh against everything else.

## Where does that operational log actually live, and how does it get written?

`.mwp/framework.log` at the product root, gitignored (it's a local operational record, not a design artifact). Every script — `scaffold.sh`, `sync.sh`, `pivot.sh`, `compact.sh`, `doctor.sh`, `registry.sh`, `status.sh` — appends exactly one line per invocation (timestamp, script name, args, exit code) via a shared `log_invocation` function in `scripts/lib/log.sh`. It's wired in via `trap ... EXIT` right after each script computes its own product root, rather than a manual logging call placed before every individual exit point — several scripts have multiple scattered failure paths, and a trap fires exactly once no matter which one actually triggers, catching success, an early usage error, and a genuine mid-script failure all the same way.

Worth knowing if you're writing a similar trap yourself: don't reference `"$?"` directly inside a trap command string that also runs a command substitution (like `$(basename "$0")`) earlier in the same argument list — the substitution's own exit status silently overwrites `$?` before it gets read, and every invocation ends up logged as `exit=0` regardless of what actually happened. This bug was caught live while testing entry `0014` — the fix is to capture `$?` into a variable as its own statement first: `trap 'EC=$?; log_invocation ... "$EC"' EXIT`. See `docs/DEVELOPMENT.md`'s script conventions and `docs/evolution/0014-minimal-operational-log.md`'s Outcome section.

## Is `scripts/*.sh` under any kind of automated test coverage, or is manual sandbox verification per entry the only check?

Both, now. Manual scratch-copy verification (per `docs/DEVELOPMENT.md`'s script conventions) is still how a change gets checked when it's first made, but as of entry `0030` there's also `tests/run_tests.sh` — a standing, dependency-free bash test harness (no `bats-core` or any other third-party framework, just bash, which every `scripts/*.sh` file already requires) covering all 7 scripts. It exists specifically so previously-caught bugs can't silently regress: the entry `0014` `$?`-trap bug and the entry `0027` sprint dead-weight-file fix are both pinned down as explicit assertions now, not just prose in an evolution entry's Outcome section. `bats-core` was the original plan (it's the standard tool the migration/testing research behind entries 0029/0030 pointed at), but it couldn't be installed in the sandbox this framework is developed in — no root for `apt`, and the sandbox's network egress blocked `git clone`, `npm install -g`, and any other registry fetch. The bash-only fallback turned out to be arguably the better fit anyway, since it doesn't add a new cross-platform install requirement to `docs/TOOLING_MATRIX.md`. `tests/` is framework-repo-only (it validates this framework's own scripts, not anything a product team would need) — it's not part of the "Using this framework for a new product" copy steps. See `docs/evolution/0030-script-test-harness.md` and `docs/DEVELOPMENT.md`'s "Testing this framework" section.

## Can I `--pivot` a sprint the same way I can pivot a feature?

No, and this is deliberate, not a gap — `scripts/pivot.sh` is hardcoded to look inside `features/`, and running it against a sprint name fails immediately (`Error: .../features/SPRINT-xxx not found`), confirmed by testing it directly. The reason isn't an oversight: Pivot/Persevere is Lean's response to a single falsifiable hypothesis being tested by one Micro-PDLC feature. A sprint under Agile-PDLC is a shared, time-boxed batch — there's no single riskiest assumption a whole sprint is testing, so "pivot the sprint" doesn't have a clean meaning the way "pivot the feature" does. `--sprint` mode is otherwise fully verified: `scaffold.sh --sprint` produces the identical 6-stage structure and correct `CONTEXT.md` substitution as `--feature` mode, and both `sync.sh` and `compact.sh` work against a sprint path unchanged (they already take a generic workspace path, not a feature-specific one). See `docs/evolution/0006-test-sprint-mode.md`.

That same testing pass did surface one real, unrelated gap, since fixed: `scaffold.sh --sprint` used to create a `FEATURE_META.md` (C-V-R scoring fields) that `registry.sh` — hardcoded to scan `features/` only — silently never read, while `scaffold.sh`'s own printed instructions misleadingly told a sprint user to fill it in and score it. Entry `0027` resolved this by skipping `FEATURE_META.md` creation entirely for `--sprint` mode rather than making `registry.sh` sprint-aware — the same reasoning as the pivot answer above: a sprint isn't a single scored hypothesis the way a feature is, so C-V-R scoring doesn't have a clean meaning applied to one. `--sprint` mode's printed next-steps now say so explicitly instead of pointing at a file and a script that would've silently ignored it.

## Is the framework itself versioned? Are individual stages versioned?

Yes to both. A root `VERSION` file gets bumped whenever an evolution entry is adopted and changes something shipped, and it's tiered — `MAJOR.MINOR.PATCH` — rather than a flat count: patch for doc-only wording fixes, minor for backward-compatible additions, major for anything that could break or silently miscopy something already scaffolded from an older version (a renamed template file, changed script argument behavior, a changed `CONTEXT.md` contract shape). Every major bump gets a row in `docs/MIGRATIONS.md` describing what broke and what to do about it; minor/patch bumps don't, since they're backward-compatible by definition — so a product team can skip straight to the latest minor/patch, and only needs to actually read something when the major number moved. Each file in `.mwp-templates/` also carries its own one-line version marker, incremented independently, since stages evolve at different rates.

This exists specifically because propagation to new products is a plain file copy, not a git submodule — the moment a product repo copies `.mwp-templates/`, it loses this repo's git history, so a version marker is the only way to later tell "what did we start from." There's still no automated tool that pulls newer templates into an existing product and merges them against that product's own changes, or that diffs a product's `VERSION` against the framework's current one — `docs/MIGRATIONS.md` is a human-readable table, not a script, and reading it is still a manual step. See `docs/evolution/0015-framework-and-template-versioning.md` for why versioning exists at all, and `docs/evolution/0029-tiered-version-scheme.md` for why it's tiered rather than a flat integer.

## If I start implementing entries from the backlog, do the other changes happen automatically, or do I need to execute each one individually?

Individually — nothing cascades. Each entry in `docs/evolution/` is implemented and marked `adopted` on its own, following the "Proposed → Adopted workflow" in `EVOLUTION_LOG.md`: pick one (by Overall Rank, normally), run its stepwise plan, record the outcome, update its status, update the log table row. Finishing one entry never automatically triggers, half-implements, or invalidates another, even entries that are related.

"Related" is worth being careful with, though: entry `0020`'s "Known cross-entry collisions" section documents cases where two entries touch the same file (`0018` and `0012` both edit all six stage `CONTEXT.md` files) or where one depends on another's output existing first (`0016`'s status script depends on `0009`'s `SYNC_LOG.md`). Overall Rank already sequences around most of that, but before starting an entry it's worth a quick check of whether anything it depends on is still unimplemented — the rank alone doesn't encode dependencies, just relative priority.

## Are the framework's constraints actually enforced, or just stated as instructions?

Mostly yes now, for the checks that have been adopted so far. Shared-schema collisions (`0005`) are mechanically checked: `sync.sh` greps a feature's `04_architecture_design/outputs/` and `05_development_test/src/` against a declared `shared_path:` list in `.mwp/GLOBAL_CONTEXT.md` and refuses to sync on a match, auto-writing a pre-filled `BLOCKED_REASON.md`. Token guardrails (`0004`) are also checked now, but as a warning, not a hard block: `sync.sh` estimates each output file's tokens (`word_count × 1.3`, a heuristic, not a real tokenizer) against the ceiling declared in that stage's `CONTEXT.md`, and prints a warning if it's over — the sync still proceeds either way. The single most foundational constraint — that an agent only reads the files explicitly declared in its stage's `CONTEXT.md` — now has a real, if deliberately imperfect, check: every stage's `outputs/` includes a self-reported `Context_Manifest.md` (every file the agent actually read), and a human cross-checks it against the declared `READ ONLY` scope at the same Obsidian review gate everything else goes through. Extras are a flag, not an automatic block. **None of these three checks is airtight** — the manifest is self-reported, not OS-level traced; the shared-path check only catches a literal name match; and the token check is a word-count estimate applied per-file everywhere even though the stage `CONTEXT.md` wording itself isn't fully uniform (per-file vs. per-stage vs. per-named-file). All three limitations are stated plainly rather than hidden. Heavier versions of each (file-access tracing, a more thorough collision detector, a real tokenizer) are noted next steps only if the lighter version proves insufficient in practice — deliberately not built until that's a concrete need. See `docs/evolution/0018-scope-containment-verification.md`, `docs/evolution/0005-automate-shared-schema-collision-check.md`, and `docs/evolution/0004-enforce-token-guardrails.md`.

## Does the framework monitor active features, and does that happen per-stage?

Yes to both, but only as an on-demand check, not a running/alerting system — there's no daemon or service in this framework for a monitor to live in. `scripts/status.sh` scans all active features (and sprints) on request and reports two views from one scan: per-workspace (current stage, blocked or not, which stage it's blocked at, last-sync timestamp from `SYNC_LOG.md`) and a stage-level rollup (how many workspaces are sitting at, or blocked at, each of the 6 stages). The stage-level view is what surfaces systemic problems — if features keep stalling at the same stage, that's a signal the stage's contract or tooling needs attention, not that each feature independently had bad luck. It shares its `features/*/` and `sprints/*/` scanning logic with `registry.sh` via `scripts/lib/scan_features.sh`, rather than each script walking the folder tree its own way. See `docs/evolution/0016-status-monitoring-script.md`.

No alerting (Slack pings, "notify if blocked > N days") is planned — same reasoning as no-auto-install (`0013`) and no-configurable-logging (`0014`): building a watcher implies a running system this framework doesn't have, and there's no concrete need for one yet. See `docs/evolution/0016-status-monitoring-script.md`.

## How do I find out who approved a feature's stage transitions, and when?

`cat <feature_path>/SYNC_LOG.md`. Every time `scripts/sync.sh` successfully advances a stage, it appends one line — timestamp, from-stage, to-stage, and an approver if one was passed as the optional 4th argument (`./scripts/sync.sh <feature_path> <from> <to> [approver]`). No approver given just logs `unspecified`. This is deliberately a lightweight paper trail, not an auth system — the approver field is freeform text, nothing validates who's actually running the command. `SYNC_LOG.md` lives inside the feature's own folder under `features/`, so it's covered by the same `.gitignore` rule as the rest of a workspace's live data — it doesn't get committed to this framework repo, only to whatever product repo the feature actually lives in. See `docs/evolution/0009-sync-audit-trail.md`.

## How do I avoid re-testing an idea that already failed in an earlier feature?

Check `LESSONS_LEARNED.md` at the product root before writing a new feature's riskiest assumption — stage `01_discovery_ideation`'s `CONTEXT.md` now requires it as reading before that stage does its work. The file is populated automatically by `scripts/pivot.sh` on every `--pivot`: it distills one row per killed feature (assumption tested, why it failed, date, archive path) from that feature's `Riskiest_Assumption.md` and, if the feature made it that far, its `Validation_Report.md` — read *before* the rest of the feature (stages 03–06) is permanently purged, since that's otherwise the only copy of that outcome. If either source file is missing, the row gets a clear placeholder instead of failing the pivot. Like the other generated files in this framework, it's append-only — never hand-edit a past row. See `docs/evolution/0008-lessons-learned-register.md`.

## How does a stage record an incidental discovery (a rate limit, a library footgun) for future features, if it can't write outside its own `outputs/`?

It doesn't write the shared file directly — it writes a plain-text `./outputs/Learnings_Note.md` (one discovery per line) inside its own declared scope, and a script performs the actual cross-boundary append. For stages 01–05, `sync.sh` folds each non-empty, non-comment line into the product-root `LEARNINGS.md` (as a `| Date | Feature | Discovery |` table row) at that stage's normal transition. Stage 06 is terminal — nothing ever syncs past it — so `pivot.sh` handles it instead, folding stage 06's `Learnings_Note.md` in on either `--pivot` (before the purge) or `--persevere`.

This split exists because the more obvious design — telling each stage's agent to append to `LEARNINGS.md` directly — was tried first and rejected: `LEARNINGS.md` sits outside every stage's `outputs/`, and the framework's write-containment rule has no "except as declared" exception for writes the way the read rule does for `READ ONLY` references. A direct write would have quietly violated the same load-bearing rule entry `0018` built a whole verification mechanism around. Routing it through a script instead mirrors the pattern entry `0008` already established for `LESSONS_LEARNED.md`: the agent never leaves its sandbox, the script does the actual cross-boundary write. See `docs/CONSTRAINTS.md`'s "Cross-boundary writes go through a script" entry and `docs/evolution/0012-shared-learnings-file.md`'s Outcome section.

## Are there any known bugs in the framework's own documentation, not just gaps in what's built?

One was found and fixed: figuring out the correct relative path for a new stage-01 reference to `LESSONS_LEARNED.md` required actually counting directory depth (a scaffolded stage lives 3 levels below product root, at `features/<name>/<stage>/`, confirmed directly against `scaffold.sh`'s own directory creation), and that count didn't match the `../../` (2 levels) every existing stage `CONTEXT.md` was using for references back to product-root files like `GLOBAL_CONTEXT.md` and `CRITICAL_ESCALATION.md` — one level short. Logged as entry `0025` rather than silently fixed at the time (it touched 6 files and deserved confirmation, not just one person's arithmetic), then confirmed and corrected once ranked into the backlog: all 6 stage `CONTEXT.md` files now use `../../../` for these references, spot-checked by actually `cd`-ing into a sandboxed scaffold and resolving each path with `ls`.

The general lesson holds regardless: these relative paths are prompt text an LLM agent reads and interprets, not something any script mechanically resolves — so a wrong path here doesn't fail loudly, it just costs an agent a bit of self-correcting exploration each time, silently, indefinitely, until someone happens to count levels by hand or a tell-tale inconsistency (like two different depths for the same kind of reference inside one file) gets noticed. Worth double-checking path depth explicitly — by counting real scaffolded folders, not by pattern-matching neighboring lines — any time a new cross-file reference is added to a stage `CONTEXT.md`. See `docs/evolution/0025-relative-path-depth-audit.md`.

## What happens to a `BLOCKED_REASON.md` once a human resolves it — does it get deleted?

No, never delete it — always move it to `<feature_path>/.escalations_archive/`. This wasn't always the rule: the original wording in `CRITICAL_ESCALATION.md` allowed a human to "delete or archive" it, and deleting is the path of least resistance under time pressure, which quietly erases exactly the failure-mode history that would help the next feature avoid repeating the same mistake. `scaffold.sh` now creates `.escalations_archive/` empty at every feature/sprint's setup, so it's always there waiting. It's a plain accumulating folder — nothing prunes it, and no helper script manages it (deliberately not built yet, same reasoning as `0013`'s no-auto-install and `0014`'s no-configurable-logging: no concrete need has shown up). It never needs its own `.gitignore` entry or `repomix`/`graphify` exclusion either, since it lives under `features/<name>/`, which the repo's `.gitignore` already ignores wholesale, and every `repomix`/`graphify` invocation in this framework is already scoped to a single subdirectory (stage 05's `./src/`, stage 04's own folder), never the feature root where the archive sits. See `docs/evolution/0010-archive-not-delete-escalations.md`.

## Does the framework check for accidentally-hardcoded secrets or credentials before they sync forward?

Yes, as of entry `0034`. `sync.sh` scans every stage's `outputs/` on every sync (not just specific stage transitions — a secret can show up in raw discovery notes just as easily as in stage 05 code) for a private-key block, an AWS-shaped access key ID, a credential-shaped filename (`.pem`, `.key`, `id_rsa`, etc.), or a generic `api_key`/`secret`/`password`/`token` assignment with a non-trivial value. It explicitly ignores `{{PLACEHOLDER}}`-style values, since that's this framework's own template convention and would otherwise false-positive on every `.mwp-templates/` file. Unlike the token-guardrail check (`0004`, warns only), this one blocks the sync outright and auto-writes `BLOCKED_REASON.md` naming exactly what was found — a leaked credential is a different order of consequence than an oversized file, since it can't be un-leaked once it propagates forward or gets committed to the product's real repo.

Same honesty as every other mechanical check in this framework: it's pattern-based, not a real secret scanner (no entropy analysis, no provider-side key validation), so it catches the common, careless shapes of a leak, not every possible one. See `docs/evolution/0034-secrets-credential-guardrail.md` and `docs/CONSTRAINTS.md`'s "Side-effects & escalation" section.

## `git push` was rejected with "GH013: Repository rule violations... Push cannot contain secrets" — but I didn't add a real secret. What happened?

This actually happened once already, in `tests/test_secrets_guardrail.sh` (entry `0034`'s own test suite): one fixture used a Stripe-shaped fake value (`sk` + `_live_` + 24 chars) to trigger the generic-credential-assignment check, and GitHub's push protection flagged it as a real Stripe API key — because `sk_live_` is Stripe's actual live-secret-key prefix, and GitHub's scanner matches on that shape regardless of whether the rest of the string is real. The irony: a test written to prove the framework's own secrets guardrail works tripped GitHub's *equivalent* guardrail on the way out.

The fix has two parts, in order:
1. **Edit the file** so the fixture no longer matches any real provider's key format — for a generic-assignment test like this one, there's no need for a provider-shaped prefix at all; a plainly-fake value (`not-a-real-secret-1234567890`) still exercises the same detection code path.
2. **Fix the commit, not just the working tree.** If `git commit` already ran before the push was attempted, the flagged string is baked into that commit object — editing the file and committing again leaves the *first* commit (with the secret) still in history, and GitHub will keep rejecting the push as long as that commit is included. If it's the most recent, not-yet-pushed commit, `git add <fixed file> && git commit --amend --no-edit` replaces it cleanly; if the secret is buried in an older already-local commit, that needs an interactive rebase instead of a simple amend.

Worth remembering for any future test fixture meant to look like a real credential (this framework will likely need more of them, given `0034`): a synthetic value with no recognizable provider prefix (Stripe's `sk_live_`/`sk_test_`, AWS's `AKIA`, GitHub's `ghp_`, Slack's `xox[abp]-`, Google's `AIza`, etc.) tests the same generic detection logic without risking a real scanner — this framework's own or GitHub's — treating it as a live leak.

## If I re-run `sync.sh` for a transition I already ran, or call `pivot.sh --persevere` twice, does anything break?

Not anymore, as of entry `0038`. It used to: `Learnings_Note.md`'s fold-into-`LEARNINGS.md` step had no way of knowing it had already run, so a second invocation for the same transition (or a second `--persevere`) re-appended the exact same discovery line(s) again, silently duplicating rows in what's supposed to be a once-per-discovery register. Fixed by renaming `Learnings_Note.md` to `Learnings_Note.md.folded` immediately after a successful fold — same archive-not-delete reasoning entry `0010` already established for `BLOCKED_REASON.md`, so the content is still there to inspect, just marked as already-processed. A second run then finds nothing to fold and skips it cleanly.

Worth noting what's *not* affected: `SYNC_LOG.md` still gets a new line on every `sync.sh` invocation, re-run or not, and that's intentional — it's a literal audit trail of when the script actually ran, not a once-per-discovery register the way `LEARNINGS.md` is. The bug was specific to `Learnings_Note.md`'s folding logic, not to sync's core copy behavior (which was already idempotent — re-copying the same files over themselves is harmless).

## Does `registry.sh` check that `FEATURE_META.md`'s `c`/`v`/`r` values actually make sense?

Yes, as of entry `0039`. Before this, a non-numeric or out-of-range value (someone typing "high" instead of a number, or a stray typo) silently computed as a `SCORE` of `0` — `awk` coerces a non-numeric string to `0` in arithmetic context — indistinguishable from a feature that's genuinely, legitimately the lowest priority in the queue. Now, any feature whose `c`/`v`/`r` are all present but at least one isn't a valid integer `1`–`5` (the scale `docs/PRIORITIZATION_GUIDE.md` documents) gets routed to a new "Malformed (needs attention)" section in the generated registry instead, naming exactly which field(s) are bad and their actual value. A feature with a *missing* field (blank, not yet filled in) still goes to "Not yet scored" as before — that's a different, legitimate state from a present-but-garbage value, and the two are now distinguished rather than conflated.

This is the runtime-data counterpart to entry `0033`'s template-contract tests: `0033` checks that the *templates* match the patterns the scripts parse; `0039` checks that what a human actually *fills in* is valid, not just present.

## Is there anything that stops a real secret from ending up in a git commit to this repo (or a product repo), not just a product's `sync.sh` transitions?

Yes, optionally — `hooks/pre-commit` (entry `0040`), added after a real near-miss: a test fixture in this framework's own `tests/test_secrets_guardrail.sh` used a Stripe-shaped fake key and got past a local `git commit` before GitHub's server-side push protection caught it, by which point fixing it required a `git commit --amend` rather than never having the secret in a commit at all. The hook reuses the exact same detection patterns as `sync.sh`'s entry-0034 secrets guardrail (private-key block, AWS-shaped key, generic hardcoded assignment, credential-shaped filename), but scans staged files instead of a stage's `outputs/`, and blocks the commit outright if it finds something.

Same convention as the existing `hooks/post-commit`: it's a sample file, not auto-installed. Copy it to `.git/hooks/pre-commit` and `chmod +x` it to actually turn it on, in this repo or any product repo built from it. `git commit --no-verify` bypasses it for a deliberate false positive.

## Does this framework's own repo run its test suite automatically, or only when someone remembers to?

Both, now — as of entry `0041`, `.github/workflows/test.yml` runs `tests/run_tests.sh` on every push and pull request to `main`, so a script regression shows up as a failed check on the commit/PR rather than only being caught the next time a human happens to run the suite by hand. Framework-repo-only, like `tests/` itself — meaningless without the test suite it runs, so it doesn't travel to product repos any more than `tests/` does.

Worth being upfront about: this workflow's verification from within Claude's sandbox was necessarily partial when it was added — the sandbox has no way to actually trigger a GitHub Actions run and watch it execute, only to confirm the YAML parses correctly and reason about the (already-exhaustively-tested) command it runs. Whether it actually fires and passes on GitHub itself is confirmed the first time someone pushes past this entry, not before.

## Has an external, critical-theory lens ever been applied to this framework, not just internal first-principles self-critique?

Yes, once — entry `0042`, a companion to `0001` rather than a duplicate of it. `0001` is an internal critique from the framework's own designer, stripping assumptions to find what's structurally load-bearing; it's rigorous, but it's still a same-author, same-session critique, and it has a blind spot a differently-shaped critical lens exists to catch. Entry `0042` applies that different lens — academic critical-theory criteria (argument/evidence, methodology, context/bias, limitations, significance) — to the framework's documentation and self-hosting evidence as shipped through entry `0041`.

Its core, previously unnamed finding: every one of the 203 assertions in `tests/run_tests.sh` verifies that scripts behave correctly against synthetic scratch data — none of them measure whether folder-scoped context actually produces better AI-agent output than the alternatives, which is the framework's real, untested thesis. A passing suite had been functioning, informally, as if it settled that question; it doesn't, and nothing else in this repo does either, since no real pilot has run yet (see "Is developing the framework itself a PDLC process?" above). It also found an unacknowledged vendor lean — `CLAUDE_WORKFLOW_PLAYBOOK.md`'s Claude-surface defaults sit in tension with entry `0031`'s stated anti-lock-in policy even though the underlying mechanism (plain markdown, folder scoping) genuinely is agent-agnostic — and that every one of the 41 prior entries has been self-graded: the same agent that implements a change also writes its tests and marks it `adopted`, with no external or adversarial review at any point.

None of this triggered a rebuild of the 6-stage/C-V-R core — entry `0001`'s trigger conditions for that remain the correct gate, and this entry doesn't move them. What it did trigger: four small, doc-only corrections adopted the same day — this FAQ entry, a README pointer alongside `0001`'s, explicit wording in `tests/run_tests.sh`'s output and `docs/DEVELOPMENT.md` distinguishing "the scripts work" from "the methodology is validated," and a strengthened neutrality line at the top of `CLAUDE_WORKFLOW_PLAYBOOK.md`. See `docs/evolution/0042-critical-theory-audit.md` for the full analysis and its own trigger conditions for when to act further.

## Does tracking metrics for the upcoming pilot need an exception to the anti-bloat / no-configurable-logging / no-alerting constraints?

No — and treating it as an exception would invite the wrong thing to get built. Entry `0042`'s assumption NA4 established a concrete need (the framework has no evidence for its own actual thesis), and entry `0043` worked out where that need actually collides with `docs/CONSTRAINTS.md`'s existing rules. It doesn't, if scoped correctly: those rules specifically reject *configurability* (logging levels, per-stage overrides) and *continuous/polling* machinery (an alerting or watcher process) — the "coordination requires software" pattern entry `0001` already names. Neither rule says "don't measure anything." `.mwp/framework.log` (entry `0014`) is already a non-configurable, always-on, single-purpose log, and nobody's ever treated it as a bloat exception, because it isn't one.

So the dividing line isn't "metrics vs. no metrics," it's which specific metric needs which kind of machinery:

- **Free:** anything computable from files the framework already writes (`SYNC_LOG.md`, `framework.log`, `LESSONS_LEARNED.md`, `BLOCKED_REASON.md` timestamps) — no new code at all, just reading what's there.
- **In scope, still minimal:** a narrow, single-purpose, non-configurable addition — one more appended log line, or a one-off on-demand audit script — in the same shape as `SYNC_LOG.md` or `status.sh`. Not a metrics subsystem, just more instances of the pattern this framework already uses everywhere.
- **Out of scope:** anything that would poll, alert, or run continuously (a live dashboard, real-time telemetry). That's the actual watcher-process pattern the constraint exists to block, and there's no concrete need for one to check a single pilot on demand.

See `docs/CONSTRAINTS.md`'s "Complexity & coordination" section for the codified version of this line, and `docs/PILOT_MEASUREMENT_PLAN.md` for how it was actually applied — a prioritized, tiered list of what's free, what needs a small addition, and what's deliberately deferred.

## Has the pilot measurement plan's Tier 2 instrumentation actually been built, or is it still just a plan?

Built, as of entry `0044` — `sync.sh`'s token and secrets guardrails now append one line per real event to `<workspace>/GUARDRAIL_LOG.md`, and a new 8th script, `scripts/audit_manifest.sh`, compares a stage's self-reported `Context_Manifest.md` against its `CONTEXT.md`'s declared scope on demand. Both were dry-run against a synthetic feature (a normal sync, a deliberately oversized file, a deliberately undeclared manifest read, a deliberate secrets block that was fixed and retried) before being trusted on the real pilot — matching the same "test in a scratch copy before it's real" discipline `docs/DEVELOPMENT.md` already asks for everywhere else.

That dry run earned its keep: cross-checking `framework.log` against `GUARDRAIL_LOG.md` for the same event surfaced a real, previously undiscovered bug in code that predates this plan — exiting from inside a bash function called with no arguments of its own (`usage()` in `scaffold.sh`, `check_secrets_guardrail()` in `sync.sh`) blanked `framework.log`'s args field, since `$*` inside a function refers to that function's own (here, empty) positional parameters, and this shadowing leaks into the `trap ... EXIT` handler. Every prior test checked these paths' exit *code*, never cross-checked the logged *args* against the real invocation, so it had never been caught. Fixed by capturing `SCRIPT_ARGS="$*"` before any function or the trap is defined, with new regression coverage in `tests/test_logging.sh`. See `docs/evolution/0044-tier2-instrumentation-and-dry-run.md`.

## Does this framework use Claude's Skills (the `SKILL.md` packages, like docx/pptx/xlsx/pdf), or is that the same thing as `CLAUDE.md`'s "Automated skill routing" section?

Different mechanisms that happen to share a word. `CLAUDE.md`'s "Automated skill routing" section routes to the CLI tool stack in `docs/TOOLING_MATRIX.md` — Fabric, Graphify, Repomix, DuckDB — text-processing tools invoked before or during a stage's own work. ## Does this framework support delegating work to sub-agents within a stage, or running multiple agents in parallel on the same feature?

Not built, and deliberately so for now — not an oversight. A sub-agent delegation pattern (a bounded research sub-agent inside a stage, say, or parallel agents working different Decision Nodes as sketched in `0001`'s section 3) would only be worth building once real pilot friction actually shows up around this — and none has yet, since no pilot has run.

Revisit against `0001`'s own trigger conditions (listed at the end of that entry) if and when they actually fire, not before.
