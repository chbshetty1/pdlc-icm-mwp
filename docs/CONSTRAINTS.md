# Framework Constraints

Every hard, non-negotiable rule this framework enforces or asks an agent to follow — in one place, instead of scattered across `IDENTITY.md`, `CLAUDE.md`, and individual evolution entries. Documentation only: nothing here changes behavior, it just makes the existing rules discoverable from a single file.

Each entry states the rule, why it exists, and where it's actually declared or enforced (a template file, a script, or an evolution entry with the full reasoning).

## Scope & containment

- **An agent only reads/writes within its active stage's declared scope.** Never reads files outside the explicit `READ ONLY` list in that stage's `CONTEXT.md`, never writes outside that stage's `outputs/` (or `src/`/`tests/` in `05_development_test/`). This is the framework's single most load-bearing rule — everything else (token discipline, parallel-feature safety) depends on it holding. See `.mwp-templates/IDENTITY.md` ("Directory containment"), `CLAUDE.md`'s "Structural constraints."
- **Scope compliance is self-reported, not mechanically traced.** Every stage produces a `Context_Manifest.md` (every file the agent says it read), cross-checked by a human against the contract's declared scope at the same review gate as everything else. Extras are a flag, not an automatic block — some may be legitimate and just under-declared. This makes containment *visible and auditable* rather than purely trust-based, but it is **not a hard technical guarantee**: an agent could in principle read outside scope and omit it from the manifest. If self-reporting proves unreliable in practice, the next step is real file-access tracing (e.g. wrapping execution with `strace`/equivalent) — deliberately not built yet, only as a future entry if a concrete need shows up. See `docs/evolution/0018-scope-containment-verification.md`.
- **Shared/global state has exactly one owner.** A feature's `04_architecture_design` or `05_development_test` stage never edits a shared schema or component directly from its isolated context — declare it in `.mwp/GLOBAL_CONTEXT.md`'s `shared_path:` list instead, and route the change through a human or a dedicated Core Data Anchor feature. Mechanically checked (not just convention): `sync.sh` greps declared shared paths against `04_architecture_design/outputs/` and `05_development_test/src/` at the relevant transitions and refuses to sync on a match. Not airtight — only catches a literal path/name match. See `docs/CLAUDE_WORKFLOW_PLAYBOOK.md`'s "Shared Architecture Sync," `docs/evolution/0005-automate-shared-schema-collision-check.md`.
- **Cross-boundary writes go through a script, never directly through an agent.** Any file outside a stage's own `outputs/` that a stage still needs to append to (`LESSONS_LEARNED.md`, `LEARNINGS.md`) is never written by the agent directly — the write-containment rule above has no "except as declared" exception the way the read rule does. Instead the agent writes a plain note inside its own `outputs/`, and `pivot.sh` or `sync.sh` performs the actual cross-boundary append as a mechanical step. See `docs/evolution/0008-lessons-learned-register.md`, `docs/evolution/0012-shared-learnings-file.md` (the latter's Outcome section documents this pattern being chosen specifically because a more direct design would have violated the rule above).

## Side-effects & escalation

- **No silent tool auto-install.** Installing software is a bigger side effect than anything else this framework does. A missing tool (Fabric, Graphify, Repomix, DuckDB, Mermaid CLI) escalates via the standard `BLOCKED_REASON.md` contract with the exact install command — a human decides whether to run it. `doctor.sh --install-missing` exists as an explicit, narrow opt-in (three tools, single non-interactive commands only), never the default. See `docs/evolution/0013-verify-tooling-matrix.md`.
- **Escalate, don't loop.** If a stage's validation or test step fails twice consecutively for the same reason, stop — don't attempt a third automated fix. Write `BLOCKED_REASON.md` per the template in `CRITICAL_ESCALATION.md` and end the session. See `.mwp-templates/IDENTITY.md` ("Escalate, don't loop"), `.mwp-templates/CRITICAL_ESCALATION.md`.
- **Resolved escalations are archived, never deleted.** Once a human resolves a `BLOCKED_REASON.md`, it moves to `<feature_root>/.escalations_archive/` — deleting it would erase failure-mode history a later feature could otherwise learn from. See `docs/evolution/0010-archive-not-delete-escalations.md`.
- **A stage output that looks like a hardcoded credential blocks the sync outright.** `sync.sh` scans every stage's outputs for a private-key block, an AWS-shaped access key ID, a credential-shaped filename, or a generic `api_key`/`secret`/`password`/`token` assignment with a non-trivial value (excluding obvious `{{PLACEHOLDER}}` values), on every sync — not just specific transitions. Unlike the token-guardrail warning below, this blocks and auto-writes `BLOCKED_REASON.md`, the same severity as the shared-path collision check above: a leaked secret can't be un-leaked once it syncs forward or gets committed. Mechanical, pattern-based, not a real secret scanner — not airtight. See `docs/evolution/0034-secrets-credential-guardrail.md`.

## Complexity & coordination

- **No configurable logging.** Whatever operational log this framework ships (see `docs/evolution/0014-minimal-operational-log.md`) is single always-on, no levels, no per-stage overrides — a configurable logging subsystem is the same "coordination requires software" machinery entry `0001` identifies as the pattern this framework deliberately avoids, and it would add surface area that context-bundling tools have to be told to ignore.
- **No alerting or watcher process.** Nothing in this framework polls, notifies, or runs continuously — `status.sh` is checked on demand, when a human wants to know, not something that pings on a timer. There's no running service for a watcher to live in, and building one would be the same category of overreach already rejected for auto-install and configurable logging. See `docs/evolution/0016-status-monitoring-script.md`.
- **Narrow, on-demand measurement isn't the same thing as configurable logging or alerting, and isn't excluded by the two rules above.** The two rules above target *configurability* (levels, per-stage overrides) and *continuous/polling* machinery — not measurement itself. A single-purpose, non-configurable log appended by an existing script (matching `SYNC_LOG.md`'s shape), or a one-off script that computes a metric on demand from files that already exist, stays within what this framework already does elsewhere. What would cross the line: a configurable metrics subsystem, a dashboard, or anything that polls or runs continuously — that's the same "coordination requires software" pattern being rejected, just with a metrics label on it. See `docs/evolution/0043-pilot-measurement-plan.md` for where this line was actually tested against a concrete case.
- **Generated files are never hand-edited.** `.mwp/FEATURE_PRIORITY_REGISTRY.md` (from `registry.sh`), `LESSONS_LEARNED.md` (appended by `pivot.sh`), and `<feature>/SYNC_LOG.md` (appended by `sync.sh`) are all regenerated or appended by their owning script — treat them as read-only output, edit the source (`FEATURE_META.md`, a feature's own outcome files) instead. See `docs/evolution/0003-computed-priority-registry.md`, `0008-lessons-learned-register.md`, `0009-sync-audit-trail.md`.
- **Token guardrails are declared, checked, and advisory.** Every stage `CONTEXT.md` states a max-token ceiling per output file; `sync.sh` estimates actual usage (`word_count × 1.3`, a heuristic, not a real tokenizer) and warns if it's over — it never blocks the sync. Treat a warning as a signal to trim before the next stage builds on it. See `docs/evolution/0004-enforce-token-guardrails.md`.

## Extensibility & composability

- **New capability, not new coupling.** When a proposed change would only be useful to a subset of products built from this framework, or requires meaningfully different machinery than the folder/`CONTEXT.md` model already has, prefer shipping it as something separate (a standalone script, a companion tool, a plugin a product opts into) over folding it into the framework's core. The core stays the smallest thing that makes the 6-stage/`CONTEXT.md` model work. Reviewed at evolution-entry time, not mechanically enforced — a judgment call, same as the rest of this section. See `docs/evolution/0031-composability-anti-lock-in-principle.md`.
- **No proposal may hard-couple the framework to one vendor, model, or proprietary format.** Every existing dependency already satisfies this (plain markdown; any LLM agent per the folder/`CONTEXT.md` contract, `docs/FAQ.md`'s first entry; free/open-source tools in `docs/TOOLING_MATRIX.md`) — this makes that property explicit and reviewable rather than an accident of what got chosen. A new, single-vendor standard needs either broad multi-vendor adoption or a concrete, currently-unmet need before this framework takes it on as a dependency, however well-marketed it is at the time. Both rules restate truths entry `0001` already identified (rejecting "coordination requires software" as unexamined dogma; markdown being correct because it's the cheapest thing satisfying two hard constraints, not because of any specific vendor) — see `docs/evolution/0031-composability-anti-lock-in-principle.md`.

## Why this file exists, and what it isn't

None of the rules above are new — this file doesn't change any of them, it just collects what was previously only discoverable by reading `IDENTITY.md`, `CLAUDE.md`, and several individual evolution entries separately. If a future change alters one of these rules, update the entry here in the same commit as the evolution entry that changes it (same discipline as `EVOLUTION_LOG.md`'s own append-only convention — don't let this file drift out of sync with the thing it's summarizing). See `docs/evolution/0017-consolidated-constraints-doc.md`.
